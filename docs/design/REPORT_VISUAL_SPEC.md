# FirstLight Report Visual Design Specification

**Version:** 1.0.0
**Last Updated:** 2026-01-26
**Companion Document:** `DESIGN_SYSTEM.md`

---

## Overview

This specification defines the visual design for FirstLight's emergency intelligence reports. All designs reference the Design System for colors, typography, and components.

**Report Types:**
1. Executive Summary (1-page)
2. Full Analysis Report (multi-page)
3. Map Products (standalone)
4. Interactive Web Report

**Target Audiences:**
- Emergency managers (primary)
- Local government officials
- Affected residents
- Media/journalists
- Technical analysts (appendix)

---

## 1. Executive Summary Report (1-Page)

### 1.1 Purpose

A single-page document that answers:
- What happened?
- How bad is it?
- Who is affected?
- What should I do?

This is the PRIMARY document for decision-makers who need information in 30 seconds or less.

### 1.2 Page Layout Specification

**Dimensions:** US Letter (8.5" x 11") or A4 (210mm x 297mm)
**Margins:** 0.5" (36px at 72dpi) all sides
**Grid:** 12-column, 16px gutters

### 3.5 Flood Extent Map Template

**Primary map for flood events:**

**Visual Hierarchy:**

1. **Flood overlay** (most prominent)
   - Colors: Flood severity palette from Design System
   - Opacity: 70-80%
   - Pattern: Solid fill with severity differentiation

2. **Critical infrastructure** (second priority)
   - Hospitals: Red cross icons, 24px
   - Schools: Blue cap icons, 20px
   - Shelters: Green house icons, 20px
   - Fire stations: Orange diamond icons, 20px
   - Labels: 10px, white halo

3. **Major roads** (contextual)
   - Highways: `#718096`, 2px width
   - Major streets: `#A0AEC0`, 1px width
   - Labels: 10px, white halo

4. **Water bodies** (background)
   - Pre-existing: `#E2E8F0`, no label
   - Rivers: `#CBD5E0`, 1px blue border

5. **Land** (base)
   - Background: `#F7FAFC`
   - Urban areas: `#EDF2F7`

**Map Extent Guidelines:**
- Include full flood extent plus 10-20% buffer
- Show nearest major city for context
- Include major highways for orientation

## 6. Technical Implementation Notes

### 6.1 HTML/CSS Framework Recommendations

**For Static Reports (PDF generation):**
- Use Paged.js for CSS-based PDF rendering
- WeasyPrint as alternative
- Maintain print stylesheet alongside screen

**For Web Reports:**
- Framework-agnostic CSS (works with any JS framework)
- CSS Custom Properties for theming
- No JavaScript required for basic functionality

### 6.2 Map Generation Tools

**Recommended Stack:**
- Matplotlib with Cartopy for static maps
- Folium for interactive web maps
- Mapbox GL JS for high-performance web maps
- QGIS for print-quality cartographic output

**Export Settings:**
- Web: PNG at 144 DPI, optimized
- Print: PNG at 300 DPI or vector PDF
- Thumbnails: PNG at 72 DPI, max 300px width

### 6.3 Responsive Breakpoints

```css
/* Mobile First */
.report { width: 100%; }

/* Tablet and up */
@media (min-width: 768px) {
  .report { max-width: 720px; margin: 0 auto; }
}

/* Desktop */
@media (min-width: 992px) {
  .report { max-width: 960px; }
}

/* Large Desktop */
@media (min-width: 1200px) {
  .report { max-width: 1140px; }
}

/* Print */
@media print {
  .report { max-width: 100%; }
  .no-print { display: none; }
}
```

### 6.4 Accessibility Checklist

For every report, verify:

- [ ] All images have descriptive alt text
- [ ] Color is not the only way to convey information
- [ ] Contrast ratios meet WCAG AA (4.5:1 for text)
- [ ] Interactive elements have visible focus states
- [ ] Document has proper heading hierarchy
- [ ] Tables have proper header markup
- [ ] Links are descriptive (not "click here")
- [ ] PDF is tagged for screen readers
- [ ] Form fields have labels (if applicable)
- [ ] No content relies solely on sensory characteristics
