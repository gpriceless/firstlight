# Value Chain Analysis: LLM Control Plane

## Summary

The proposal transforms FirstLight from a monolithic pipeline into a controllable service with four API surfaces. Priority recommendation: **Ship Phases 0-2 as MVP** -- security hardening, state externalization, and control API deliver standalone value for MAIA integration. Phase 3 (event stream) is high-value but can follow quickly. Phase 4 (OGC/STAC/Taskiq) is standards compliance and should not block partner onboarding.

Critical dependencies are correctly ordered: PostGIS schema is load-bearing for everything downstream, and security must precede any external-facing API. However, three gaps risk engineering rework: (1) the 7 research documents referenced in the proposal do not exist, (2) Taskiq is split across Phase 3 (webhook delivery) and Phase 4 (formal adoption) creating a dependency inversion, and (3) the checkpoint resume mechanism has no spec for how existing pipeline stages actually consume checkpoint payloads.

MAIA can begin integration after Phase 2 (Control API) for job submission and state observation. They need Phase 3 for real-time event push. They do not need Phase 4 unless they require OGC-compliant job submission or STAC-based result discovery.

## Dependency Graph

### Security Hardening (Phase 0)
- REQUIRES: Nothing (modifies existing auth/config code)
- ENABLES: All of Phase 1-4 (no external API without this)
- EXTERNAL: None

### State Externalization (Phase 1)
- REQUIRES: Phase 0 complete (customer_id field needed in jobs table)
- ENABLES: Control API (reads/writes PostGIS), Event Stream (LISTEN/NOTIFY on job_events), STAC Publishing (reads job data)
- EXTERNAL: PostGIS Docker image availability

### Control API (Phase 2)
- REQUIRES: Phase 0 (tenant scoping), Phase 1 (PostGIS state backend, job_events table)
- ENABLES: Event Stream (events are generated by control API writes), OGC Integration (tool schema generation reused), STAC Publishing (job completion triggers publishing)
- EXTERNAL: MAIA's agent framework readiness

### Event Stream (Phase 3)
- REQUIRES: Phase 1 (job_events table + NOTIFY trigger), Phase 2 (events to stream)
- ENABLES: Webhook delivery (same CloudEvents payload), operational dashboards (metrics endpoint)
- EXTERNAL: Partner SSE client implementation

### Webhook Delivery (Phase 3)
- REQUIRES: Event Stream (CloudEvents payload), **Taskiq broker (Phase 4 task 4.8)** -- DEPENDENCY INVERSION
- ENABLES: Asynchronous partner notifications
- EXTERNAL: Partner webhook endpoint availability

### OGC Integration (Phase 4)
- REQUIRES: Phase 2 task 2.9 (tool schema generation), Taskiq (async job execution)
- ENABLES: Standards-compliant job submission from any OGC client
- EXTERNAL: pygeoapi compatibility with FastAPI mount

### STAC Publishing (Phase 4)
- REQUIRES: Phase 1 (PostGIS/pgSTAC schema), Phase 2 (job completion data)
- ENABLES: Result discovery via standard STAC clients, dynamic map tiling via titiler
- EXTERNAL: pgSTAC schema namespace isolation (risk flagged in design doc)

## Parallelization Opportunities

- **Track 1 (State + API):** Phase 0 -> Phase 1 (tasks 1.1-1.10) -> Phase 2 (tasks 2.1-2.11). Sequential, core value chain.
- **Track 2 (Infrastructure):** Task 1.3 (PostGIS Docker image) can run in parallel with 1.1 (StateBackend ABC). Task 1.5 (enums) in parallel with 1.4 (schema migration).
- **Track 3 (Standards, deferred):** Tasks 4.1-4.4 (OGC) and 4.5-4.7 (STAC) have no mutual dependency and can run in parallel once their prerequisites ship.
- **Merge point:** Task 4.8 (Taskiq) ties webhook delivery (3.6) to the formal broker setup. This is the convergence risk.

Within Phase 2, tasks 2.2-2.9 are largely parallel after 2.1 (router setup). Within Phase 3, SSE (3.1-3.4) and webhook registration (3.5) can run in parallel after 3.2 (route package).

## Gaps

| # | Gap | Impact | Recommendation |
|---|-----|--------|----------------|
| G1 | 7 research documents cited in proposal.md do not exist at their listed paths | Critical | Either produce them before engineering starts or remove the citations. Engineers will look for them and find nothing. |
| G2 | Taskiq dependency inversion: task 3.6 (webhook delivery) uses Taskiq but task 4.8 (Taskiq setup) is in Phase 4 | Critical | Move Taskiq broker setup (4.8) to Phase 3 or early Phase 3. Webhook delivery cannot ship without the broker. |
| G3 | Checkpoint resume mechanism (state-management spec) has no spec for how pipeline stages consume checkpoint payloads | Moderate | The StateBackend ABC defines `checkpoint()` and `restore_checkpoint()` but no spec describes what the OrchestratorAgent does with the restored payload. Each pipeline stage needs a "resume from checkpoint" contract. |
| G4 | No rate limiting spec for Control API | Moderate | LLM agents can hammer the transition endpoint. Design doc lists this as an open question but the spec should define at least per-customer rate limits before shipping Phase 2 externally. |
| G5 | Phase+Status enum values inconsistent between specs | Moderate | State-management spec says phases are QUEUED/DISCOVERING/.../COMPLETE. Control API spec says DISCOVERY/INGESTION/.../COMPLETE. Tasks.md says PENDING/VALIDATING/DISCOVERING/etc. Pick one canonical list. |
| G6 | No migration path for existing SQLite job data | Moderate | DualWriteBackend handles new writes but existing historical jobs in SQLite are not migrated. If operators need continuity of job history, a data migration script is needed. |
| G7 | OGC auth boundary undefined | Moderate | Task 4.4 says "verify auth middleware does not intercept /oapi requests" but the security spec requires auth on all non-public endpoints. Which is it? pygeoapi's own auth layer is not configured anywhere. |
| G8 | STAC collection strategy is an open question | Minor | Design doc asks "one collection per event type, or per customer, or per mission?" Tasks.md assumes one per event type (task 4.6). Make this a decision, not a question. |
| G9 | No health/readiness probe exemption list | Minor | Security spec says health probes are exempt from auth but no spec defines which paths are exempt. |
| G10 | COG output requirement (STAC spec) has no connection to existing pipeline output format | Moderate | The spec requires Cloud-Optimized GeoTIFF output but does not assess whether the current pipeline already produces COGs. If it does not, this is a significant new work item hidden inside a "publishing" task. |

## Recommendations for Product Queen

- **R1: Ship Phases 0-2 as "Partner Preview" milestone.** After Phase 2, MAIA can submit jobs, read state, trigger transitions, and inject reasoning. This is a usable integration point. Do not gate partner onboarding on Phase 3 or 4. (Cites: Control API is RICE 4.0, highest standalone value.)

- **R2: Move Taskiq broker setup from Phase 4 to Phase 3.** The webhook delivery task (3.6) depends on Taskiq, but Taskiq is formally introduced in Phase 4 (4.8). This creates a dependency inversion that will block Phase 3 completion. (Cites: G2, dependency graph.)

- **R3: Resolve the Phase+Status enum naming before Phase 1 starts.** Three different naming conventions appear across specs. Engineering will waste time reconciling these during implementation if not resolved now. (Cites: G5.)

- **R4: Defer OGC integration (Phase 4 tasks 4.1-4.4) to a separate change.** RICE score is 0.2. No current partner is requesting OGC compliance. It adds pygeoapi as a dependency with mount-conflict risk for zero near-term user value. Keep it on the roadmap but out of this change. (Cites: Priority ranking, OGC row.)

- **R5: Produce or delete the 7 missing research documents.** The proposal cites them as completed research. They do not exist. Either they were never written, or they exist elsewhere. Either way, engineering cannot validate design decisions against phantom references. (Cites: G1.)

- **R6: Add a decision on rate limiting before Phase 2 ships externally.** LLM agents operating in a loop can generate hundreds of transition attempts per minute. Without rate limiting, a misbehaving agent can degrade service for all tenants. (Cites: G4.)

- **R7: Investigate COG output compatibility before committing to STAC publishing timeline.** If the pipeline does not currently produce Cloud-Optimized GeoTIFFs, the STAC publishing scope is significantly larger than the tasks suggest. (Cites: G10.)
